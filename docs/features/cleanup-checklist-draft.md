## ログ関連

### 調査用ログの残存確認

- **背景**: 過去に `_debug_correlation`(approval 相関調査)、`_debug_raw_stdin`(stdin 受信調査)、`_debug_event_payload`(payload 調査)など、**調査セッション単位で一式追加されたログ群**が消し忘れで蓄積した。Ticket 2.5 で全削除したが、同様のパターンは今後も発生しうる
- **チェック対象**: デバッグ目的で追加された一時的なログ・出力(`print`, 一時ファイル書き込み、ad-hoc な logger 呼び出しなど)
- **チェック方法**: 未定(`grep` パターンでの検出、pre-commit フック導入などを後続で検討)
- **判断軸**:
  - 運用中に障害調査で使う価値があるか → Yes なら Operational に昇格、No なら削除
  - 迷ったら削除(Operational 昇格の基準は厳しめに保つ)
- **過去実例**:
  - `_debug_correlation` 一式(approval 相関調査)
  - `_debug_raw_stdin` 一式(stdin 受信調査)
  - `_debug_event_payload` 一式(payload 調査)
  - `_debug_stage` の分岐トレース群(hook 処理経路の確認)
  - いずれも「集中的に投入され、そのまま固まって残った」パターン

### Operational Log 記録対象カタログの陳腐化

- **背景**: `docs/logging.md` のカタログは Phase2 範囲で最低限の定義から始めた。実装が進むにつれカタログと実態がズレる可能性がある。Phase2 完了後に本格的な棚卸しを行う前提
- **チェック対象**: `docs/logging.md` の「Operational Log 記録対象カタログ」節
- **判断軸**:
  - カタログ記載のイベントが実際にログ出力されているか
  - 実装で出力されているイベントがカタログに記載されているか
  - 使われていない・不要になったエントリはないか
- **頻度目安**: Phase 切り替え時、または大きな機能追加後

### TODO(state-audit-redesign) マーカーの進捗

- **背景**: Ticket 2.5 で削除した `state.*` 系ログの再構築ポイントを示すマーカー。State/Audit Log の正式設計チケット着手時に一括回収する予定
- **チェック方法**: `grep -rn 'TODO(state-audit-redesign)' src/`
- **判断軸**:
  - State/Audit 正式設計チケットが進んでいるか確認
  - 進んでいないまま長期間放置されていないか
- **想定着手時期**: Phase2 完了後

### Debug Log / Notification / StateAudit ハンドラの実装状況

- **背景**: Ticket 2.5 では `LogKind` の enum と `get_logger` の口だけ用意し、ハンドラ本実装は後続とした。各種別が「口だけある」状態が長く続くと、使う側が混乱する可能性
- **チェック対象**: Debug / Notification / StateAudit の各ハンドラの実装状況
- **判断軸**:
  - 実装されていないハンドラを呼び出しているコードがないか
  - 必要性が生まれているのに放置されていないか

## ドキュメント関連

### logging.md と実装の齟齬

- **背景**: Ticket 2.5 の議論過程で、ドキュメント(仕分けメモ)と実装の依存関係を見落とす事例が発生した(`_debug_stage` と `state.*` ログの依存関係)。人間と AI の議論を重ねても見落としは発生するという教訓

- **チェック対象**: 各種ドキュメントの「後続課題」「後続チケットの候補」節
- **判断軸**: 実施済みなら記述を更新、不要になったなら削除

## 運用ログ(Operational Log)関連

### ログレベルの実態確認

- **背景**: `logging.md` で INFO/WARN/ERROR の判断基準を「ユーザーまたは運用者の対応が必要か」と定めたが、実装時の判断が揺れる可能性
- **チェック対象**: 実際に出力されている Operational Log のレベル分布
- **判断軸**:
  - ERROR が過剰に出ていて狼少年化していないか
  - WARN/ERROR の使い分けが実態に即しているか

### ログサイズと保持期間

- **背景**: Ticket 2.5 ではローテーション・保持期間を後続課題とした。運用が進むとログファイルが肥大化する
- **チェック対象**: `logs/operational.log` 等のサイズ
- **判断軸**: ローテーション実装の必要性、保持期間の妥当性

## 設計・運用戦略関連

### 完璧主義の再確認

- **背景**: このドキュメント自体が「完璧を目指さず定期棚卸しで悪化を防ぐ」戦略に基づいて存在する。運用を続けるうちに「もっと厳密にルール化すべき」という衝動が生まれることがあるが、その前に一度立ち止まって考える必要がある
- **チェック対象**: 最近追加されたルール・ドキュメント・仕組み
- **判断軸**:
  - ルール追加によって認知コストが過剰に上がっていないか
  - 「検出と事後対応」で解決できるものを「事前防止」で解こうとしていないか
  - 新しいルールが「書いた満足感」で止まっていないか(実効性の検証)
