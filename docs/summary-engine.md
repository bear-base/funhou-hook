# summary-engine

ログを LLM で要約し、サマリー専用チャンネルに流すエンジン。

## このドキュメントの位置づけ

これは基本設計レベルの方針整理である。詳細設計・実装計画は別途進める。

## 背景

- 1行ログだけでは「エージェントが何をやっているか追いきれない」
- 1行ログの加工で意味理解を狙うのは過剰なため、別レイヤーとして
  サマリーで対応する
- 設計書 phase2 の中心機能のひとつ
- AI による要約のプロンプト試行錯誤自体に実験価値がある

## 目的

- ターン単位でエージェントの作業内容を要約し、人間が後追いで把握できる
  状態を作る
- 承認判断時に「ここまでの経緯」を示せるようにする

## やること

- ログ蓄積 → 要約 → 通知の一連を実装する
- LLM 呼び出しは差し替え可能なインターフェースにする
- 最小トリガーから始め、運用で粒度を調整する

## やらないこと(意図的な非ゴール)

- 件数・時間ベースのトリガー(まずはターン単位で十分かを検証する)
- 異常系の独立サマリー(運用で必要になったら検討)
- 進捗系の独立サマリー(同上)
- `TaskCompleted` をトリガーとして使う(`Stop` と重複しやすい。ログとして
  記録するがサマリートリガーにはしない)
- 高度なログ解析、複数エージェント統合、サマリー履歴の検索

## トリガー

最小実装として以下の2つ:

- `Stop` イベント(ターン終了時)
- `PermissionRequest` イベント直前(承認判断材料として)

`TaskCompleted` などの他イベントはトリガーとしては使わないが、
ログとしては記録されサマリーの素材になる。

運用しながら粒度を見る。うるさい/不足どちらでも調整余地あり。

## サマリー種別の考え方

サマリーは1種類に統一する(完了系・経緯系・異常系などの種別分けはしない)。

- トリガーが複数あっても、サマリーの本質は「ここまでの作業のまとめ」で
  共通
- AI に発火元のトリガー種別を渡せば、AI 側でトーンや構成を調整できる
- 種別分けで複雑化するより、AI 任せでシンプルに保つ

## AI による要約価値判定

要約する価値があるかを AI に判定させる:

- ログをプロンプトに渡し、要約に値しなければ空文字列を返すよう指示
- 空が返ってきたら通知をスキップする
- ルールベースで「短いターンをスキップ」を実装しなくて済む
- 判定基準もプロンプトで調整可能

## LLM の選定

- 初期実装: **Gemini API**(`gemini-2.0-flash` 想定)
  - 安価、無料枠あり、サマリー用途に十分な能力
- インターフェースを抽象化して差し替え可能にする
  - 例: `generate_summary(prompt: str) -> str` 形式で隠蔽
- Claude API、OpenAI、ローカル LLM への切り替えを想定

## ログの取得元

- 現状ログは `funhou.log` テキストファイルに書き出し
- サマリーエンジンはこのファイルを読み取って入力にする
- ターン単位で切り出す方針(詳細は実装時)

## サマリーの出力

- 統一メッセージ型 `SummaryMessage` に乗せる
- フィールド:
  - `message`: 要約本文
  - `next`: 次のアクション(任意、AI が判断できれば書く)
  - `trigger`: 発火元イベント(デバッグ・分析用)
- 通知先は **サマリー専用の Slack チャンネル**(1行ログとは別)

## エラーハンドリング

- LLM 呼び出し失敗、タイムアウト、不正出力などは静かに失敗する
- リトライは最大1回
- デバッグログにエラーを記録するが、Slack には通知しない
- 次のトリガーで自然に再試行のチャンスが来る

## プロンプト設計

- プロンプトは外部ファイル or 定数で管理し、差し替えやすくする
- 初期版の方針:
  - 入力: 直近ターンのログ(時系列、ツール名と内容)+ トリガー種別
  - 出力: 要約本文 / 次のアクション(任意)
  - 要約に値しない場合は空文字列を返す指示を含める

## 設定項目(`funhou.toml` 想定)

- LLM プロバイダ選択
- API キー参照(秘匿情報は `.env`)
- サマリー専用チャンネル設定
- (将来) トリガー追加・抑制の設定

## 設計判断

- **トリガーはターン単位を主軸に**: ユーザーの認知単位と一致
- **サマリーの粒度判定は AI に任せる**: ルールベースの限界を認める
- **エラーは静かに飲む**: サマリーは「あれば便利」程度、失敗が致命的
  ではない
