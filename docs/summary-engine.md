# summary-engine

ログを LLM で要約し、サマリーチャンネルに流すエンジン。

## 背景

- 1行ログだけだと「エージェントが何をやってるか追いきれない」という痛みがある
- 1行ログの加工で意味理解を狙うのは過剰なので、別レイヤーとしてサマリーで対応する
- 設計書 phase2 のメイン機能のひとつ

## 目的

- 一定区切りでエージェントの作業内容を要約し、人間が後追いで把握できるようにする
- AI による要約のプロンプト試行錯誤自体に実験価値がある

## やること

- ログ蓄積 → LLM 呼び出し → サマリー生成 → 通知の一連を実装する
- LLM 呼び出しは差し替え可能なインターフェースにする
- トリガー条件を最小から実装する

## やらないこと(意図的な非ゴール)

- 高度なログ解析(構造化抽出、依存関係分析など)
- 複数エージェントのサマリー統合
- サマリー履歴の検索・分析機能

## LLM の選定

- 初期実装は **Gemini API**(`gemini-2.0-flash` 想定)
  - 理由: 安価、無料枠あり、サマリー用途に十分な能力
- インターフェースを抽象化して差し替え可能にする
  - 例: `generate_summary(prompt: str) -> str` という形で隠蔽
  - Claude API、OpenAI、ローカル LLM などへの切り替えを想定

## ログの取得元

- 現状ログは `funhou.log` テキストファイルに書き出し
- サマリーエンジンはこのファイルを読み取って入力にする
- 直近 N 件 / 時刻範囲 で切り出す方針(詳細は実装時)

## トリガー条件

最小実装として以下を検討、絞り込みは実装時:

- ログが一定件数蓄積した時(例: 10件)
- 一定時間経過した時(例: 5分)
- タスク完了の検知
- 承認要求の直前(「ここまでの経緯」として)
- エラー発生時

連続発火を防ぐクールダウン制御は必要に応じて入れる。

## サマリーの出力

- 統一メッセージ型 `SummaryMessage` に乗せる
- フィールド: `message`(要約本文)、`next`(次のアクション、推測なので任意)
- 通知先は **サマリー専用の Slack チャンネル**(1行ログとは別)

## プロンプト設計

- 試行錯誤しやすいよう、プロンプトは外部ファイルか定数で管理
- 初期版の方針:
  - 入力: 直近のログ(時系列、ツール名と内容)
  - 出力: 何をしたか / 結果 / 次に何をするか の3要素
- 「次に何をするか」は推測になるので、自信がなければ省略させる

## 設定項目(`funhou.toml` 想定)

- LLM プロバイダ選択
- API キー参照(秘匿情報は `.env`)
- トリガー条件のしきい値(件数、時間など)
- サマリーチャンネル設定

## 未決事項

- トリガー条件の優先順位と最初に実装するもの
- ログ取得の具体的な実装(ファイル末尾追従 vs 別途バッファ)
- プロンプトの初期版テキスト
- 「タスク完了」をどう検知するか(明示シグナル? 一定時間アイドル?)
