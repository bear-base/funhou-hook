# AI エージェント分報システム 設計ドキュメント

## コンセプト

AIエージェント（Claude Code等）の作業を、**人間の部下の分報**のように監視・制御するシステム。

コンソールに張り付くのではなく、別ウィンドウやSlackを**チラ見**するだけで「今何やってるか」「次に何をさせるか」が判断できる状態を作る。

### 解決するペインポイント

| ペイン | 原因 | 解決 |
|---|---|---|
| コマンドラインの認知負荷が高い | 作業と報告が同じ場所に混在 | 報告を別チャネルに分離し、危険度で視覚的に分類 |
| 聞く/聞かないの制御が難しい | コマンド単体では文脈がわからない | タスク文脈 × ルール × AIスクリーニングで判定 |
| 作業報告がサマリーで欲しい | 生ログは量が多すぎる | 区切りごとにAIがサマリーを生成 |
| チラ見で監視したい | コンソールは別作業と併用しづらい | 任意のプラットフォームに分報を流す |

### 目指す体験

```
コンソール張り付き型（現状）
  作業 → 停止 → 確認 → 指示 → 作業 → 停止...
  （自分の作業が毎回中断される）

分報チラ見型（このシステム）
  エージェントが裏で作業 → 分報が流れてくる
  → 自分の作業の合間にチラ見
  → 必要な時だけ返信で指示
  （自分の作業フローを壊さない）
```

## コアの出力：3種類のメッセージ

システムが出力するメッセージは以下の3種類のみ。全てのアダプターはこのフォーマットを受け取る。

### log — 作業ログ

エージェントのツールコールを逐次記録する。

```json
{
  "type": "log",
  "timestamp": "2026-04-09T10:03:00Z",
  "level": "info",
  "tool": "Read",
  "target": "src/config.ts",
  "message": "Read src/config.ts"
}
```

### summary — 作業サマリー

一定の区切り（タスク完了、N件のログ蓄積、時間経過）でAIが生成する。

```json
{
  "type": "summary",
  "timestamp": "2026-04-09T10:15:00Z",
  "message": "DB接続設定をconfig.tsで変更し、テスト24件全てパス。dist/を再ビルド済み。",
  "next": "マイグレーション実行（承認待ち）",
  "log_count": 12,
  "duration_sec": 720
}
```

### approval — 承認要求

エージェントが人間の判断を必要とする場合に発行する。

```json
{
  "type": "approval",
  "timestamp": "2026-04-09T10:16:00Z",
  "level": "danger",
  "tool": "Bash",
  "command": "npx prisma migrate deploy",
  "reason": "本番DBへのマイグレーション実行",
  "await_response": true
}
```

## 危険度判定

### レベル定義

| レベル | 意味 | 通知動作 |
|---|---|---|
| `info` | 安全な操作。読み取り、テスト実行など | ログに流すだけ |
| `warning` | 注意が必要。ファイル編集、外部API呼び出しなど | メンション付きで通知、実行は止めない |
| `danger` | 承認が必要。破壊的操作、本番環境への変更など | メンション付き通知 + 承認待ちで停止 |

### 判定フロー：3段階フォールバック

```
ツールコール発生
  │
  ▼
① Hard Rules（settings.yaml）
  │ マッチした → そのレベルを適用
  │ マッチしない ↓
  ▼
② Skills / Hooks（プロジェクト固有の判断基準）
  │ 判定できた → そのレベルを適用
  │ 判定できない ↓
  ▼
③ AIスクリーニング（LLM呼び出し）
  │ タスク文脈 + コマンド内容から判定
  ▼
  レベル確定 → 通知ディスパッチ
```

### Hard Rules（例）

```yaml
rules:
  # 絶対止めるもの
  - match: "Bash(rm -rf /)"
    level: danger
  - match: "Edit(*.env*)"
    level: danger
  - match: "Bash(*deploy*|*migrate*)"
    level: danger

  # 絶対静かでいいもの
  - match: "Read|Glob|Grep"
    level: info
  - match: "Bash(npm test|npm run lint)"
    level: info
```

### Skills による文脈判定（例）

```markdown
# SKILL.md - コマンド危険度判定

## このプロジェクトの文脈
- dist/ build/ は使い捨て。削除は info
- .env, credentials は最重要。編集は danger
- テスト実行は常に info
- 本番系コマンド（deploy, migrate）は常に danger

## 判断に迷ったら
- 破壊的 + 復元困難 → danger
- 破壊的 + 復元容易 → warning
- 読み取り専用 → info
```

### タスクコンテキストによる上書き

同じコマンドでもタスクによって危険度が変わる。タスク単位でルールを上書きできる。

```yaml
tasks:
  build-cleanup:
    overrides:
      "Bash(rm *)": info       # 掃除タスクなら気にしない

  db-migration:
    overrides:
      "Bash(*)": warning        # 全コマンド注視
      "Bash(*migrate*)": danger # マイグレーションは必ず止める
```

### 動的なルール変更

通知チャネル経由でルールをその場で変更できる。

```
ユーザー: @AI これは #info にして
→ 当該操作の危険度がセッション中 info に変更

ユーザー: @AI これは #danger
→ 当該操作の危険度がセッション中 danger に昇格
```

履歴が蓄積されることで、AIスクリーニングの判断精度が向上する（学習効果）。

## サマリー生成

### トリガー条件

サマリーは以下のタイミングで自動生成される。

- タスクまたはサブタスクの完了時
- ログが一定件数（例: 10件）蓄積した時
- 一定時間（例: 5分）経過した時
- エラー発生時
- 承認要求の直前（「ここまでの経緯」として）

### サマリーの構成

```
📋 10:03-10:15 まとめ
config.tsのDB接続設定を変更し、テスト24件パス。dist/を再ビルド済み。
次: マイグレーション実行（承認待ち）
```

サマリーに含める要素：

- **何をしたか**（作業内容の要約）
- **結果**（成功/失敗、テスト結果など）
- **次に何をするか**（次のアクション）

## 通知API

### 設計方針

通知先（プラットフォーム）はアダプターとして差し替え可能にする。コアはプラットフォームを知らない。

### アダプター一覧

| アダプター | 用途 | 実装コスト |
|---|---|---|
| **stdout**（別ターミナル） | 最小プロトタイプ。`tail -f` で監視 | 最小 |
| **Slack webhook** | チーム利用、チラ見に最適 | 小 |
| **OS通知** | アラート（danger）だけ飛ばす | 小 |
| **WebSocket → ブラウザ** | リッチなWeb UI | 中 |
| **VSCode サイドバー** | 開発中に自然 | 中 |
| **メニューバー常駐** | macOSでチラ見 | 中 |

### アダプターの設定（例）

```yaml
channels:
  terminal:
    enabled: true
    output: "/tmp/ai-bunhou.log"   # tail -f で監視
    levels: [info, warning, danger]

  slack:
    enabled: true
    webhook: "https://hooks.slack.com/..."
    channel: "#ai-bunhou"
    mention_on: [warning, danger]
    mention_to: "@you"
    levels: [info, warning, danger]

  os_notification:
    enabled: true
    levels: [danger]               # 承認待ちだけ
```

### 双方向通信（返信の受け取り）

通知先からの返信を受け取る口を用意する。ただし全てのアダプターが双方向である必要はない。

| アダプター | 方向 | 返信でできること |
|---|---|---|
| stdout | 一方向 | — |
| Slack | 双方向 | 承認/拒否、軽い指示、ルール変更 |
| OS通知 | 一方向 | — |
| Web UI | 双方向 | 承認/拒否、指示、詳細確認 |

Slackに求める操作は**軽いものだけ**に限定する：

- 承認 / 拒否
- 次の指示（一言）
- 危険度の変更
- 「今何やってる？」「diff見せて」

複雑な操作（詳細なdiff確認、方針の大幅変更など）はWeb UIやコンソールで行う。

## 実装ロードマップ

### Phase 1: 最小プロトタイプ

目標：別ターミナルで分報が流れる状態を作る。

- [x] PreToolUse / PostToolUse フックの実装
- [x] ログフォーマッターの実装（色付き・絵文字）
- [x] ファイル出力（`tail -f` で監視）

実装見送り（余力があればやる）

- [ ] Hard Rules による危険度判定：本機能としての重要度が低いため先にPhase2へ

成果物：`tail -f /tmp/funhou.log` で分報が見える

### Phase 2: 通知とサマリー

目標：Slackで分報が見え、サマリーが生成される。

- [ ] Slack webhook アダプター
- [ ] サマリー生成エンジン（LLM呼び出し）
- [ ] サマリーのトリガー条件実装

実装見送り（余力があればやる）

- [ ] OS通知アダプター（danger のみ）：クロスプラットフォーム対応が重いため

### Phase 3: 双方向通信

目標：Slackから軽い指示が出せる。

- [ ] Slack Bot（受信側）の実装

## 設計原則

1. **コアはプラットフォームに依存しない** — 出力は統一フォーマット、表示はアダプター任せ
2. **全部流して見せ方を変える** — ログの量を減らすのではなく、重要度で視認性を変える
3. **判定はルール → Skills → AI のフォールバック** — 確実なものは高速に、曖昧なものはAIに
4. **人間の作業フローを壊さない** — チラ見で状況把握、必要な時だけ介入
5. **カスタマイズが前提** — 危険度の基準は人・チーム・プロジェクトで異なる
6. **段階的に育てる** — 最小構成（別ターミナル）から始めて、必要に応じて拡張
