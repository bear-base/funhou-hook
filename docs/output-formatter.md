# output-formatter

`FunhouMessage` を通知先向けの文字列に整形するモジュール。

## 背景

- 既存の `formatter.py` は terminal の `tail -f` 用に作られている
- 同じフォーマットを Slack 等のリッチテキスト環境に流すと可読性が低い
  - 重要情報(状態・ツール種別)とコマンド詳細がフラットに並ぶ
  - 長いコマンドが目立ってしまい、本当に見たい情報が埋もれる
- 環境ごとに使える表現(絵文字、太字、等幅、ANSI 色など)が異なるため、
  共通フォーマットでは限界がある

## 目的

- 環境別フォーマッタを用意し、それぞれの環境で **流し見しやすい表示** を実現する
- コマンドを「短くする」のではなく、**重要情報を目立たせ、詳細情報を背景化する**
  ことで視認性を改善する
- 意味理解はサマリー側に任せ、ここでは行わない

## やること

- フォーマッタを環境別に分離する
  - terminal 用(既存維持)
  - ファイル出力用(funhou.log 用、プレーンで良い)
  - Slack 用(新規、装飾を活用)
- Slack 用フォーマッタで以下の整形を行う
  - 構造化された `LogMessage` のフィールドごとに装飾を割り当てる
  - フルパスを `cwd` 基準の相対パスに短縮

## やらないこと(意図的な非ゴール)

- AI による意味整形(コスト・レイテンシに見合わない)
- Bash コマンドの構造解釈(`&&` 分解、ヒアドキュメント解釈)
- コマンド名のホワイトリスト整形
  - 理由1: hook が渡す `tool_input.command` はコマンド文字列まるごとで、
    subcommand 分解はされない。自前パースは原理的に不確実
  - 理由2: 危険度判定層と機能が被る
  - 理由3: subcommand 分解は本来 Claude Code 側の責務領域
- 文字数 truncate(コマンド本体に効かせると意味喪失するため)
- 必要になったら、その時点で再設計する

## 設計判断

- **環境別フォーマッタを認める**: 「リッチ通知共通」では実態に合わない
  (Slack はリッチテキスト、terminal は ANSI、file はプレーン)
- **コアの構造化メッセージは環境非依存を保つ**: 設計書の原則「コアはプラットフォーム
  非依存」は、`LogMessage` 等の型構造で担保する
- **Slack フォーマッタは消せる前提で作る**: 不要だと判断したら削除できる構造に
- **既存の terminal フォーマッタには手を入れない**: 動いているものを壊さない
- **装飾は段階的に育てる**: 初期は最小、運用で調整

## 装飾の指針(例、実装で調整可)

Slack 用フォーマッタでの装飾割り当ての例:

| フィールド     | 装飾                   | 理由                                     |
| -------------- | ---------------------- | ---------------------------------------- |
| `level`        | 絵文字                 | 状態が一目で分かる、視覚記号として最強   |
| `tool`         | 太字                   | 重要情報、目立たせる                     |
| `target`       | インラインコード(等幅) | パスやコマンド本体、詳細情報として背景化 |
| 補足メッセージ | プレーン               | 文脈情報、目立たせない                   |

絵文字は `level` のみに使う方針(`tool` には使わない、絵文字が並ぶと
うるさいため)。

これらは初期の指針であり、運用で見直してよい。

## モジュール構成案

- `src/funhou_hook/formatter.py` に追加
  - 既存の `format_message` は terminal 用としてそのまま残す
  - 新規: `format_for_slack(message: FunhouMessage) -> str`
  - 必要に応じてファイル出力用フォーマッタを切り出す
- 通知アダプタ(Slack 等)は対応するフォーマッタを呼ぶ

## 構造化メッセージの前提

本フォーマッタは、`LogMessage` の各フィールド(`level`, `tool`, `target`,
`message`)が **構造化された状態で埋まっている** ことを前提とする。

現状の実装ではフィールドが十分に埋まっていない可能性がある。フィールド充実
(hook → `LogMessage` の変換層の整備)は本ドキュメントのスコープ外だが、
本フォーマッタを機能させる前提として **対応が必要**。

## 検証方法

- 実運用で2週間程度使ってみて、以下を主観評価する
  - Slack を見る回数が減らないか(=見る気を失っていないか)
  - 「詰まってないか」の判断が一目でできるか
  - 重要情報が目立ち、詳細情報が流せる構造になっているか
- 不足を感じたら装飾を足す、過剰を感じたら削る
