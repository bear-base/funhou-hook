# output-formatter

`FunhouMessage` を通知先向けの文字列に整形するモジュール。

## 背景

- 現状の `formatter.py` は terminal での `tail -f` 想定で作られている
- これを Slack に流すと可読性が低い(フルパス、長いコマンド、フォーマット不揃い)
- ただし「軽めに情報を得たい」という設計思想があり、過剰な加工は避けたい

## 目的

- Slack でチラ見しやすい1行ログを実現する
- 「詰まってないか」が分かるレベルの可読性を担保する
- 意味理解はサマリー側に任せ、ここでは行わない

## やること

- terminal 用と Slack 用でフォーマッタを分離する
- Slack 用フォーマッタで以下の軽い整形を行う
  - 状態を絵文字で統一表示(待ち / 実行中 / 完了 / 失敗 など)
  - フルパスをワークスペースルートからの相対パスに短縮
  - 一定文字数を超えたコマンドの末尾省略
  - よく使うコマンドの簡易ホワイトリスト整形(git, npm など、必要に応じて段階的に追加)

## やらないこと(意図的な非ゴール)

- AI による意味整形(コスト・レイテンシに見合わない)
- Bash コマンドの構造解釈(`&&` 分解、ヒアドキュメント解釈など)
- 網羅的なホワイトリスト辞書化(保守負荷が無限に増える)
- これらが必要になったら、その時点で再設計する

## 設計判断

- **Slack フォーマッタは消せる前提で作る**: 不要だと判断したら削除できる構造にする
- **既存の terminal フォーマッタには手を入れない**: 動いているものを壊さない
- **ホワイトリストは最小から始める**: 運用で不足を感じたら足す
- 設定はコード直書きで開始(将来 `funhou.toml` に出すかは運用判断)

## モジュール構成案

- `src/funhou_hook/formatter.py` に Slack 用関数を追加
  - `format_for_slack(message: FunhouMessage) -> str`
- 既存の `format_message` は terminal 用としてそのまま残す
- もしくは `format_message` を `format_for_terminal` にrenameして役割を明示する(任意)

## 検証方法

- 実運用で2週間程度使ってみて、以下を主観評価する
  - Slack を見る回数が減らないか(=見る気を失っていないか)
  - 「詰まってないか」の判断が一目でできるか
  - 過剰加工で逆に情報が落ちていないか
- 不足を感じたらホワイトリストを足す、過剰を感じたら削る
