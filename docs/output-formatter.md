# output-formatter

`FunhouMessage` を通知先向けの文字列に整形するモジュール。

## このドキュメントの位置づけ

これは基本設計レベルの方針整理である。具体的な装飾の選定や実装計画は
別途進める。基本設計が固まる前に詳細・実装計画に進まないこと。

## 背景

- 既存の `formatter.py` は terminal の `tail -f` 用に作られている
- 同じフォーマットを Slack 等のリッチテキスト環境に流すと、状態・対象・
  詳細がフラットに並んで認知負荷が高い
- 環境ごとに使える表現(絵文字、太字、等幅、ANSI 色など)が異なるため、
  共通フォーマットでは限界がある

## 目的

- Slack 等で **人間の認知負荷を下げる** 表示を実現する
  - 「視線を流すだけで状態が目に入る」状態
  - 「詳細は読み飛ばしていい」と視覚的に分かる状態
- 「コマンドを短くする」「内容を分かりやすくする」ことは目的ではない
- 意味理解はサマリー側に任せ、ここでは行わない

## 設計の中核方針

### 構造化フィールドを活かす

`LogMessage` の構造(`level`、`tool`、`target`、`message`)を、
表示上の優先順位に対応付ける:

1. `level` を最初に拾えるようにする(状態が目に入る)
2. `tool` で何の操作かが分かる
3. `target` で対象がはっきりする
4. `message` は補足、必要なときに読める

`message` を主役にせず、構造化フィールドを組み立てて表示する。

### 装飾は「読み手への扱い方シグナル」

装飾の役割は、内容を変えることではなく、読み手に「どう扱うか」を
伝えること:

- 目立たせる装飾 → 「ここを最初に見る」「重要」のシグナル
- 等幅・コードブロック → 「詳細情報、興味なければ飛ばせる」のシグナル

内容を変える加工は本モジュールでは行わない。

### 意味判断はしない

formatter は **意味判断に踏み込まない**。

- コマンドが危険か、何のコマンドか、どんな目的か、を判断しない
- 渡された `level` と各フィールドを、装飾を付けて表示するだけ
- 危険度判定や意味解釈はコアの別レイヤーの責務

## やらないこと(意図的な非ゴール)

- AI による意味整形(コスト・レイテンシに見合わない)
- Bash コマンドの構造解釈(`&&` 分解、ヒアドキュメント解釈)
- コマンド名のホワイトリスト整形(コマンド同定が原理的に不確実)
- 文字数 truncate(コマンド本体に効かせると意味喪失)
- パスやコマンドの内容に基づく分岐処理(意味判断に踏み込むため)

これらが必要になったら、その時点で再設計する。

## 環境別フォーマッタ

通知先によって使える表現が異なるため、フォーマッタは環境ごとに分ける:

- **terminal 用**: 既存維持、`tail -f` での参照を想定
- **ファイル出力用**: プレーンテキスト、`funhou.log` 用
- **Slack 用**: リッチテキスト表現を活用

コアの `LogMessage` 構造は環境非依存を保つ。フォーマッタ層が環境差を
吸収する。

将来 Discord、OS 通知などが必要になれば、別フォーマッタとして追加する。

## 装飾の手段

視覚的な目立たせ方の手段は多様にある:

- 装飾系: 文字スタイル(太字、斜体)、フォント(等幅)、色
- 記号系: 絵文字、コードブロック、引用記法
- 配置系: 改行、インデント、区切り

`level` ごとに、どの手段を使うかを柔軟に決める。手段は単一に縛らず、
組み合わせを許容する。

具体的な装飾の組み合わせは **実物の Slack 表示で試作・比較して決める**
領域であり、本ドキュメントでは固定しない。

## 構造化メッセージの前提

本フォーマッタは、`LogMessage` の各フィールド(`level`、`tool`、
`target`、`message`)が **構造化された状態で埋まっている** ことを
前提とする。

現状の実装ではフィールドが十分に埋まっていない可能性がある。フィールド
充実(hook → `LogMessage` の変換層の整備)は本ドキュメントのスコープ外
だが、本フォーマッタを機能させる前提として **対応が必要**。

## 設計判断

- **環境別フォーマッタを認める**: 「リッチ通知共通」では実態に合わない
- **コアの構造化メッセージは環境非依存を保つ**
- **Slack フォーマッタは消せる前提で作る**: 不要だと判断したら削除できる
  構造に
- **既存の terminal フォーマッタには手を入れない**: 動いているものを
  壊さない
- **装飾は段階的に育てる**: 初期は最小、運用で見直す
- **完璧を目指さない**: 「使うのが嫌にならない」レベルで合格、細かい
  識別性などは運用で気になったら直す

## 進め方

1. 基本設計(本ドキュメント)
2. 詳細設計: 具体的な装飾の試作、Slack 上の見え方を比較して採用案を決定
3. 実装計画: 既存コードとのギャップ整理、変更範囲・順序の決定
4. 実装

基本設計が固まる前に詳細・実装計画に進まない。仕様が固まらないうちに
実装計画を立てると、仕様変更で計画が破綻する。
