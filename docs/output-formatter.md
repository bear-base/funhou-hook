# output-formatter

`FunhouMessage` を通知先向けの文字列に整形するモジュール。

## このドキュメントの位置づけ

これは基本設計レベルの方針整理と、現時点で採用済みの Slack 表示仕様を
記録するドキュメントである。今後の装飾追加や表示ルール変更は、基本設計
の責務分離を壊さない範囲で段階的に行う。

## 背景

- 既存の `formatter.py` は terminal の `tail -f` 用に作られている
- 同じフォーマットを Slack 等のリッチテキスト環境に流すと、状態・対象・
  詳細がフラットに並んで認知負荷が高い
- 環境ごとに使える表現(絵文字、太字、等幅、ANSI 色など)が異なるため、
  共通フォーマットでは限界がある

## 目的

- Slack 等で **人間の認知負荷を下げる** 表示を実現する
  - 「視線を流すだけで状態が目に入る」状態
  - 「詳細は読み飛ばしていい」と視覚的に分かる状態
- 「コマンドを短くする」「内容を分かりやすくする」ことは目的ではない
- 意味理解はサマリー側に任せ、ここでは行わない

## 設計の中核方針

### 構造化フィールドを活かす

`LogMessage` の構造(`level`、`tool`、`target`、`message`)を、
表示上の優先順位に対応付ける:

1. `level` を最初に拾えるようにする(状態が目に入る)
2. `tool` で何の操作かが分かる
3. `target` で対象がはっきりする
4. `message` は補足、必要なときに読める

`message` を主役にせず、構造化フィールドを組み立てて表示する。

### 装飾は「読み手への扱い方シグナル」

装飾の役割は、内容を変えることではなく、読み手に「どう扱うか」を
伝えること:

- 目立たせる装飾 → 「ここを最初に見る」「重要」のシグナル
- 等幅・コードブロック → 「詳細情報、興味なければ飛ばせる」のシグナル

内容を変える加工は本モジュールでは行わない。

### 意味判断はしない

formatter は **意味判断に踏み込まない**。

- コマンドが危険か、何のコマンドか、どんな目的か、を判断しない
- 渡された `level` と各フィールドを、装飾を付けて表示するだけ
- 危険度判定や意味解釈はコアの別レイヤーの責務

## やらないこと(意図的な非ゴール)

- AI による意味整形(コスト・レイテンシに見合わない)
- Bash コマンドの構造解釈(`&&` 分解、ヒアドキュメント解釈)
- コマンド名のホワイトリスト整形(コマンド同定が原理的に不確実)
- 意味理解を目的とした truncate(内容を分かりやすくするための要約)
- パスやコマンドの内容に基づく分岐処理(意味判断に踏み込むため)

Slack の表示制約に対応するための機械的な `target` 省略は、意味判断ではなく
表示調整として扱う。省略ルールは後述の「検証済みの事実」に従う。

## 環境別フォーマッタ

通知先によって使える表現が異なるため、フォーマッタは環境ごとに分ける:

- **terminal 用**: 既存維持、`tail -f` での参照を想定
- **ファイル出力用**: プレーンテキスト、`funhou.log` 用
- **Slack 用**: リッチテキスト表現を活用

コアの `LogMessage` 構造は環境非依存を保つ。フォーマッタ層が環境差を
吸収する。

将来 Discord、OS 通知などが必要になれば、別フォーマッタとして追加する。

## 装飾の手段

視覚的な目立たせ方の手段は多様にある:

- 装飾系: 文字スタイル(太字、斜体)、フォント(等幅)、色
- 記号系: 絵文字、コードブロック、引用記法
- 配置系: 改行、インデント、区切り

`level` ごとに、どの手段を使うかを柔軟に決める。手段は単一に縛らず、
組み合わせを許容する。

具体的な装飾の組み合わせは **実物の Slack 表示で試作・比較して決める**
領域であり、本ドキュメントでは固定しない。

## 構造化メッセージの前提

本フォーマッタは、`LogMessage` の各フィールド(`level`、`tool`、
`target`、`message`)が **構造化された状態で埋まっている** ことを
前提とする。

現状の実装ではフィールドが十分に埋まっていない可能性がある。フィールド
充実(hook → `LogMessage` の変換層の整備)は本ドキュメントのスコープ外
だが、本フォーマッタを機能させる前提として **対応が必要**。

## 実装反映

### 現状の課題

- Slack の `LogMessage` 表示で、長い `target` が inline code に入ると横に
  伸び、通知として読みづらい。
- `target` が長い場合、単純な三点リーダーだけでは省略されたことが分かり
  にくい。
- 承認待ちの `ApprovalMessage` で `command` 全文を表示すると、理由より
  コマンド詳細が目立ち、承認判断のノイズになる。
- `message` に `tool target` が含まれる場合、Slack 上で同じ対象が何度も
  表示される。

### 検証済みの事実

- Slack formatter は [../src/funhou_hook/slack_formatter.py](../src/funhou_hook/slack_formatter.py)
  に実装されている。
- `LogMessage` の Slack 表示は、`level` アイコン、`tool`、`target`、補足
  `message` の順で組み立てる。
- `level` アイコンは以下を使う。
  - `info`: `🔹`
  - `warning`: `⚠️`
  - `danger`: `⚡`
  - `error`: `🚨`
- 短い一行 `target` は inline code として表示する。
- 複数行または省略後に改行を含む `target` は code block として表示する。
- `target` が `TARGET_TRUNCATE_LIMIT` を超える場合は、全ツール共通で
  `TARGET_TRUNCATE_HEAD` 文字、`（中略：N文字）`、`TARGET_TRUNCATE_TAIL`
  文字の形に省略する。
- `N` は `len(target) - TARGET_TRUNCATE_HEAD - TARGET_TRUNCATE_TAIL` で
  算出する。バイト数や表示幅ではなく Python の文字数として扱う。
- `message` が `Completed {tool} {target}` など、すでに表示済みの target を
  含む既知の結果文の場合、Slack 補足では target 部分を繰り返さない。
- `ApprovalMessage` は `command` 全文を表示せず、`理由` と `操作`(`tool`)
  のみを表示する。
- 現在の挙動は [../tests/test_slack_formatter.py](../tests/test_slack_formatter.py)
  で固定されている。

### 未検証の仮説

- `TARGET_TRUNCATE_LIMIT = 160`、`TARGET_TRUNCATE_HEAD = 100`、
  `TARGET_TRUNCATE_TAIL = 40` は初期値であり、実際の Slack 運用で適切かは
  未検証である。
- `（中略：N文字）` は、三点リーダーより省略箇所を認識しやすい想定だが、
  長期利用で目立ちすぎないかは未検証である。
- `ApprovalMessage` で `command` を出さない方針は通知の認知負荷を下げる
  想定だが、承認判断に command 全文が必要なケースがどの程度あるかは
  未検証である。

### 修正方針

- formatter は今後も意味判断を持たない。ツール種別、コマンド内容、JSON
  構造に応じた分岐は追加しない。
- 長い `target` の全文が必要な場合は、Slack 通知ではなく元ログを参照する
  前提にする。
- Slack 上で承認待ちの詳細が不足する場合でも、formatter で command を
  解釈して短い対象を作らない。必要なら hook から `ApprovalMessage` を生成
  する層で、構造化フィールドを追加する。
- 表示の微調整は、`TARGET_TRUNCATE_LIMIT` / `TARGET_TRUNCATE_HEAD` /
  `TARGET_TRUNCATE_TAIL` の定数変更とテスト更新で行う。

## 設計判断

- **環境別フォーマッタを認める**: 「リッチ通知共通」では実態に合わない
- **コアの構造化メッセージは環境非依存を保つ**
- **Slack フォーマッタは消せる前提で作る**: 不要だと判断したら削除できる
  構造に
- **既存の terminal フォーマッタには手を入れない**: 動いているものを
  壊さない
- **装飾は段階的に育てる**: 初期は最小、運用で見直す
- **完璧を目指さない**: 「使うのが嫌にならない」レベルで合格、細かい
  識別性などは運用で気になったら直す

## 進め方

1. 基本設計(本ドキュメント)
2. 詳細設計: 具体的な装飾の試作、Slack 上の見え方を比較して採用案を決定
3. 実装計画: 既存コードとのギャップ整理、変更範囲・順序の決定
4. 実装

基本設計が固まる前に詳細・実装計画に進まない。仕様が固まらないうちに
実装計画を立てると、仕様変更で計画が破綻する。
